*会提问*其实不是一件很自然的事。直接把问题丢给模型，等它回答，这样的效果很可能不如人意。同一个模型，问题换一种写法，输出质量可能就好很多。

所以我们就需要所谓的提示词工程，也就是prompt engineering。


## 一、什么是提示词工程

提示词工程可以理解为：**一种结构化的、清晰的自然语言，能够最没有歧义地表述你的需求**

最简单的提示可能只有一句话：

```text
天空是什么颜色？
```

模型通常能回答出来。但如果任务稍微复杂一点，比如要你总结一段文字、提取指定字段、按固定格式输出、控制语气和风格，那么仅仅“提问”就不够了。

例如下面这两个提示：

```text
介绍一下抗生素。
```

```text
请用一句话向高中生解释抗生素是什么，并说明它为什么不能用来治疗病毒感染。
```

第二个提示比第一个好，不是因为用了某个神奇词汇，而是因为它更明确地告诉模型：

- 任务是什么
- 受众是谁
- 输出长度是什么
- 解释重点是什么

这正是提示词工程的核心：**减少模糊，增加有效约束。**

## 二、一个好提示通常包含什么

一个比较完整的提示通常由下面几个部分组成。

### 1. 指令

告诉模型你希望它做什么。

例如：

- 请总结这段文字
- 请将下面内容翻译成中文
- 请从文本中提取人名和地点
- 请判断这条评价的情绪倾向

### 2. 上下文

给模型补充背景信息，让它更容易理解任务。

例如：

- 目标读者是没有技术背景的产品经理
- 你是一名客服助手，需要保持礼貌和简洁
- 下文是一篇医学科普文章，不要输出未经证实的结论

上下文不是越多越好，而是越相关越好。无关背景只会稀释重点。

### 3. 输入数据

也就是你真正要模型处理的内容。

例如：

```text
文本：我觉得这家餐厅味道一般，但服务很好。
```

### 4. 输出要求

这是实际工作里最容易被忽略，但最能提升稳定性的部分。

你可以明确要求：

- 输出语言
- 输出格式
- 输出长度
- 输出字段
- 是否允许解释

例如：

```text
请输出 JSON，字段为 sentiment 和 reason。
```

或者：

```text
请仅输出地点名称，使用逗号分隔，不要添加额外解释。
```

## 三、从零样本到少样本

有一个比较常见的概念是，零样本和少样本。

### 零样本提示

不给例子，直接描述任务。

```text
判断下面句子的情绪是正面、负面还是中性：
“这个产品能用，但没有惊喜。”
```

对于很多基础任务，模型已经能完成得不错。

### 少样本提示

如果你对输出格式、标签命名、判定标准有更细要求，就可以给模型先看几个示例。

```text
请判断情绪：

文本：这个功能太好用了。
情绪：正面

文本：体验很差，页面还崩了。
情绪：负面

文本：这个产品能用，但没有惊喜。
情绪：
```

少样本提示的价值主要在于：

- 帮模型对齐你的输出格式
- 帮模型理解你的标签定义
- 帮模型靠近你真正想要的风格

少样本能够解决一类痛点就是，“模型大致懂任务，但输出总不够像你想要的结果”。一般大家喜欢在这里加长文本描述，但效果反而不如一两个例子来得好。

## 四、模型参数会影响什么

除了提示内容本身，很多平台还允许调整模型参数。常见的有：

### 1. Temperature

`temperature` 决定输出的随机性。

- 低一点：更稳定，更保守，更适合事实型任务
- 高一点：更多样，更发散，更适合创意任务

经验上：

- 摘要、问答、信息提取：适合较低温度
- 创意写作、标题生成、头脑风暴：可以稍高一些

### 2. Top P

`top_p` 也是控制采样范围的参数。

它和 `temperature` 都会影响输出多样性，通常没有必要两个一起大幅调整。很多时候只调一个就够了。

### 3. Max Length

它控制模型最多生成多少 token。

这个参数常用于：

- 控制输出长度
- 避免模型写得太散
- 控制推理和调用成本

### 4. Stop Sequences

如果你想让模型在某些标记处停止输出，可以设置停止序列。

例如让模型只列 5 条清单，或者在某个字段之后结束。

### 5. Frequency Penalty / Presence Penalty

这两个参数都与*重复*有关：

- `frequency penalty` 更关注某个词重复出现的次数
- `presence penalty` 更关注某个词是否已经出现过

一般只有在你发现模型很啰嗦时才需要明显调整它们。

## 五、设计提示的几个通用原则

### 1. 从简单开始，再逐步加约束

不要一开始就写一个超长、超复杂、层层嵌套的提示。那样很难判断到底是哪一部分在起作用。

更稳的做法是：

1. 先写最小可用版本
2. 看输出哪里不符合预期
3. 再针对性补充约束

这和调试程序很像。你不是一开始就写完美代码，而是逐步逼近正确结果。

### 2. 具体比模糊更有效

例如下面这个提示并不够清楚：

```text
解释一下提示词工程，简短一点，不要太复杂。
```

“简短一点”到底是多少？“不要太复杂”对谁来说不复杂？

更好的写法是：

```text
请用 2 到 3 句话向没有编程背景的大学生解释什么是提示词工程。
```

这时候，模型更容易命中你的预期。

### 3. 尽量说“要做什么”，而不是只说“不要做什么”

下面这种提示经常会失败：

```text
不要太啰嗦，不要问我问题，不要输出多余内容。
```

因为它没有给模型一个积极、明确的目标。

更好的写法是：

```text
请直接给出 3 条建议，每条不超过 20 个字，不要附加解释。
```

前者是在设限，后者是在定义输出。

### 4. 拆任务，别把一切压进一个提示里

如果一个任务同时要求：

- 先读懂长文
- 再抽取信息
- 再做情绪分析
- 再输出表格
- 再生成建议

那么一次性全塞进去，往往不稳定。

更好的方式是分阶段：

1. 先总结核心内容
2. 再提取结构化字段
3. 最后生成面向人的结果

复杂任务拆成多个简单提示，通常比一个超大提示更可靠。

## 六、几个典型任务怎么写提示词

### 1. 文本概括

一个基础摘要提示可以这样写：

```text
请用一句话总结下面这段内容的核心观点：

<正文>
```

如果你想让摘要更实用，可以进一步指定：

- 面向谁总结
- 保留什么重点
- 不要包含什么

例如：

```text
请用一句话向产品经理总结下面文章的核心结论，不要复述细节。
```

### 2. 信息提取

信息提取最关键的不是“能不能提”，而是“提出来后能不能稳定落到固定格式里”。

例如：

```text
请从以下文本中提取公司名和地点。

输出格式：
公司：...
地点：...

文本：
<正文>
```

如果你只说“帮我提取重要信息”，模型可能会自由发挥；如果你指定字段，它通常会稳定得多。

### 3. 问答

问答类任务最重要的是加上下文边界。

例如：

```text
请根据下列上下文回答问题。
如果上下文中没有答案，就回答“无法根据提供信息确定”。

上下文：
<材料>

问题：
<问题>
```

这类写法能减少模型“脑补答案”的概率。

### 4. 文本分类

分类任务对标签格式尤其敏感。

比如你想要输出固定标签：

```text
请将下列句子分类为：正面、负面、中性。
仅输出标签，不要解释。

句子：
这个功能勉强能用。
```

如果你发现模型输出“比较中立”“略偏正面”这种不规范答案，就说明你的输出约束不够强，需要补充示例或明确“仅允许以下标签”。

### 5. 对话与角色设定

提示词工程并不只用于单轮任务，也常用于控制模型在对话中的身份和语气。

例如：

```text
你是一名 AI 学习助手。
你的回答要清晰、耐心、避免术语堆砌。
面向对象是刚入门的大学生。
```

这样的角色说明，不是让模型“扮演人格”，而是帮助它稳定输出某种风格。

## 七、一个更实用的提示模板

刚开始用大模型，如果不熟悉的话，可以先用下面这个模板：

```text
任务：
<你要模型做什么>

上下文：
<必要背景信息>

输入：
<要处理的文本或数据>

输出要求：
1. <格式要求>
2. <长度要求>
3. <风格要求>
4. <如果不确定时怎么回答>
```

例如：

```text
任务：
总结一段产品访谈记录。

上下文：
读者是产品经理，需要快速理解用户的核心抱怨。

输入：
<访谈内容>

输出要求：
1. 用中文输出
2. 控制在 3 条以内
3. 每条不超过 30 字
4. 仅保留用户问题，不给解决方案
```


## 八、持续迭代

形成一套迭代方法不断提高效果很重要，一般可以：

- 先确定任务目标
- 再写最小可用提示
- 观察输出偏差
- 补充更精确的约束
- 用示例做格式对齐
- 通过参数调节稳定结果

不建议把它当成一次性任务，而是持续性的去解决它。

## 最后

之前我其实也觉得*模型越强，提示越不重要*。实则更多地用AI后感觉恰恰相反，模型越强，就越有机会通过更好的提示把它变成真正可用的工具。现在的模型代际差异越来越大，一些老的模型连听懂用户指令、遵守用户规则都做不到，体感上还得是用新不用旧。

对初学者来说，其实做好下面几点就够了：

- 把任务写清楚
- 把格式说清楚
- 把边界说清楚
- 学会用示例校准结果
- 学会根据任务调整参数

做到这些，至少我们就入门提示词工程啦~
